Strategy Pattern - Replace Conditional

ANTES

1 classe com múltiplos if/else

Difícil de estender

Lógica misturada

Viola OCP

Duplicação de código

DEPOIS

Interface + implementações

Fácil adição de estratégias

Testável independentemente

Segue OCP e SRP

Factory para criação

Quando usar Strategy:

• Múltiplos if/else com comportamentos diferentes

• Algoritmos que podem variar em runtime

• Necessidade de extensibilidade sem modificação

Decorator Pattern - Move Embellishment

ANTES

1 classe com múltiplas responsabilidades

Flags booleanas

Combinação limitada

Viola SRP

Difícil manutenção

DEPOIS

Interface + decorators

Combinação flexível

Responsabilidade única

Segue SRP e OCP

Composição dinâmica

Quando usar Decorator:

• Funcionalidades opcionais combinadas

• Evitar explosão de subclasses

• Adicionar comportamentos dinamicamente

Princípios SOLID Atendidos pelas Refatorações
✓
SRP (Single Responsibility): Cada classe tem uma única responsabilidade bem definida
✓
OCP (Open/Closed): Aberto para extensão (novas estratégias/decorators), fechado para modificação
✓
DIP (Dependency Inversion): Depende de abstrações (interfaces), não de implementações concretas
✓
Benefício geral: Código mais limpo, extensível, testável e alinhado aos princípios SOLID
